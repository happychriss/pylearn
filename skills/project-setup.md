# Project Setup

## 1. Folder Structure

Ensure these folders exist at the workspace root:

```
/workspace/
├── requirements.md  # ROOT — project type pointer + functional specification for all features
├── external-docs/   # Raw external material — reference docs, API specs, vendor docs (read-only)
├── knowledge/        # Curated learned notes — confirmed working config derived from docs + experiment
├── skills/          # Reusable Claude skills (this file lives here)
└── src/             # Source projects
```

Create any missing folders. Do not create a README in `docs/` or `knowledge/` — the files are self-describing.

---

## 2. Knowledge Flow

This is the rule for where project knowledge lives:

```
external-docs/  →  knowledge/           →  memory.md  +  auto-memory (~/.claude/projects/)
(reference)        (confirmed config)      (thin pointers)  (cross-session facts)
```

### external-docs/
- Drop zone for external material: API docs, reference markdown, vendor docs
- Never edit these files
- When the user adds a file here, analyse it and create or update the corresponding `knowledge/` file

### knowledge/
- Single source of truth for confirmed, component-level facts
- Knowledge learned from working on this project, mistakes made and how they were fixed

### memory.md
- Stays concise: pointers to `knowledge/` files + project-level gotchas
- Never duplicate detail that belongs in `knowledge/`

### auto-memory (`~/.claude/projects/`)
- Managed automatically by Claude's memory system across sessions
- Stores user preferences, feedback, and project context that doesn't belong in the codebase
- Do not duplicate here what already lives in `knowledge/`

---

## 3. Analysing a New Doc

When the user adds a file to `external-docs/`:

1. Read it fully
2. Extract: key concepts, configuration, API details, any quirks or caveats
3. Create `knowledge/<topic>.md` with structured notes
4. Mark unconfirmed values clearly: `# unconfirmed — from docs`
5. Tell the user what was captured and what needs validation

## 4. Container Environment Fixes

These one-time fixes are needed when the dev container is freshly started or rebuilt. Check if they're needed before assuming the environment is broken.

### Locale warning (`setlocale: LC_ALL: cannot change locale`)

The `en_US.UTF-8` locale is not generated by default in this container image. Fix:

```bash
sudo locale-gen en_US.UTF-8
sudo update-locale LANG=en_US.UTF-8
```

Verify with: `locale -a | grep en_US` — should show `en_US.utf8`.

### pnpm not on PATH

pnpm is managed by corepack but not linked by default. Fix:

```bash
sudo corepack enable
```

### PostgreSQL not running (local dev)

PostgreSQL is installed but needs to be started manually on each container boot:

```bash
sudo pg_ctlcluster 17 main start
```

If the `pylearn` database doesn't exist yet (fresh container):

```bash
sudo -u postgres psql -c "CREATE USER ubuntu SUPERUSER;"
sudo -u postgres createdb -O ubuntu pylearn
DATABASE_URL="postgresql://ubuntu@/pylearn?host=/var/run/postgresql" pnpm --filter @workspace/db run push
```

### fly CLI not on PATH in current shell

The PATH is set in `~/.zshrc` and `.env.local` is sourced there too. For the current bash/shell session, load manually:

```bash
export PATH="$HOME/.fly/bin:$PATH"
set -a && source /workspace/src/.env.local && set +a
```

## 5. Write-Back Rule

At the end of any session where behaviour was confirmed experimentally:
1. Update the relevant `knowledge/` file with the confirmed values
2. Trim any duplicate detail from `memory.md` — keep only the pointer and project-level gotchas
3. Save user preferences and non-code facts to auto-memory (`~/.claude/projects/`) — not to `knowledge/`
